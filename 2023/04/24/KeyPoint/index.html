继承中的静态变量处理

在继承中，静态成员的处理有以下几种情况：

  1. 基类静态成员变量在派生类中仍为基类的静态成员变量。派生类通过类名或者对象名访问基类的静态成员变量时，将直接使用基类的静态成员变量。也就是说，派生类中不会产生一个新的静态成员变量。
  2. 基类静态成员函数在派生类中仍为基类的静态成员函数。派生类通过类名或者对象名调用基类的静态成员函数时，将直接调用基类的静态成员函数。
  3. 派生类中可以定义自己的静态成员变量和静态成员函数。派生类的静态成员变量和静态成员函数与基类的静态成员变量和静态成员函数是相互独立的，它们互不影响。
  4. 如果派生类中定义了与基类同名的静态成员变量或静态成员函数，并且它们的类型也相同，则会隐藏基类的静态成员变量或静态成员函数。这时候，如果想要访问基类的静态成员，可以通过作用域解析运算符(::)来访问。例如，基类中定义了一个静态成员变量a，派生类中也定义了一个同名的静态成员变量a，那么可以通过Base::a来访问基类的静态成员变量a。

什么是虚继承

虚继承是C++中的一种继承方式，在多重继承中由于有多个父类，可能会产生二义性或者浪费内存等问题。虚继承可以解决这些问题，避免出现多个父类的同名成员或者重复继承的问题。

虚继承的实现原理是：在继承体系中的类继承虚基类时，不会直接继承虚基类的成员，而是继承一个指向虚基类的指针，虚基类的成员只存储在虚基类中，而不是在派生类中。

需要注意的是，虚继承不能确保消除所有二义性，只是限制了虚基类的访问路径，但在派生类中若存在非虚基类同名成员，则仍会发生二义性。此外，虚继承也可能会带来额外的运行时开销。因此，在使用虚继承时需要注意权衡和考虑实际需要。

虚基类和抽象类的关系

  1. 虚基类

虚基类是C++中的一种继承方式，在多重继承中通过使用虚继承可以避免产生二义性或者浪费内存的问题。虚基类是一种抽象的概念，它主要的特点是：

  • 被虚继承的基类叫做虚基类；
  • 虚基类在派生类对象中只有一份副本；
  • 所有对虚基类的访问都通过这个副本进行。
  1. 抽象类

抽象类是C++中面向对象编程的概念之一，它是一种特殊的类：它不能被实例化，只能用来作为其他类的基类，它的主要特点是：

  • 包含至少一个纯虚函数的类就是抽象类；
  • 抽象类不能实例化；
  • 派生类必须实现基类的纯虚函数才能够实例化。
  1. 关系

虚基类和抽象类是两个不同的概念，但是在实践中，虚继承经常用于表示一种概念上的抽象关系。比如，在一个抽象类的派生类中，如果多个继承路径上都有相同的基类，我们可以通过虚继承来避免多次继承造成的二义性或者冗余问题。

因此，虚继承可以被看作一种用于实现抽象类的技术，而抽象类则是一种设计模式，用于实现某种抽象概念。虚继承可以解决多继承中的一些问题，而抽象类则是面向对象编程中的重要概念之一，用于表示一些抽象的概念，可以提高程序的可扩展性和灵活性

static赋初值

在C++中，我们可以在定义静态变量或静态成员变量时为其指定初始值。有两种方式可以实现静态变量或静态成员变量的初始化：

  1. 在定义时直接赋值

这种方式可以直接给静态变量或静态成员变量赋值，不需要再用别的函数或语句去初始化它，比较方便。

  1. 在类外使用作用域解析运算符指定初始值

这种方式可以在类外用作用域解析运算符指定静态成员变量的初始值，也比较方便，不需要再定义静态成员初始化函数了。

无论哪种方式，只要为静态变量或静态成员变量指定了初始值，就会在程序运行时进行初始化。需要注意的是，静态变量或静态成员变量的初始化只会在程序中出现的第一个实例化对象中进行，在后续实例化对象中不会再次初始化。

类内可以定义静态成员变量吗

需要注意的是，类内定义静态成员变量时，并不会为该成员变量分配内存空间，只是告诉编译器这个静态成员变量存在，并且在程序执行时需要分配内存空间。需要在类外部定义和初始化该静态成员变量后，才会真正分配内存空间。

如果在类外定义并初始化了静态成员变量后，分配空间是在编译阶段吗

不是，在类外定义和初始化了静态成员变量后，分配空间和初始化是在程序的运行阶段进行的。

一般情况下，编译器会在编译时给静态变量或静态成员变量分配内存空间，但是对于类的静态成员变量来说，所需的内存空间大小可能取决于用户的设定，此时编译器无法预先确定所需要的内存空间大小，因此会在程序运行阶段才为静态成员变量分配内存。

具体来说，当程序启动时，操作系统为程序分配一块内存，包括了全局变量、全局常量、静态变量、静态常量、代码段等等。在其中，静态变量和静态常量存放在数据段或者BSS段（未初始化的全局变量或静态变量会被存放在这个段中），内存分配和初始化是由运行时库在程序运行时动态完成的。

所以，即使在类外定义和初始化静态成员变量，程序在运行时仍然需要进行内存分配和初始化。

什么时候需要向前声明类模板和函数模板

向前声明类模板和函数模板是为了解决模板类型依赖问题，通常在以下情况下需要向前声明类模板和函数模板：

  1. 在多个文件中使用同一模板类或函数：如果在多个源文件中使用同一个类模板或函数模板，那么在使用之前需要先向前声明该模板。这是因为模板定义通常包含在头文件中，当多个源文件都包含同一个头文件时，实际引入的模板定义会出现重复，导致编译器无法判断这些定义的真正有效性。

异常

  1. 先用throw抛出异常，再用tyr和catch捕获，通常抛出多少异常就要进行多少次捕获。除了作用域e就销毁了，所以本质上捕获到的是一个临时变量。这个拷贝的临时对象会在被catch以后销毁。
  2. 异常是通过抛出对象而引发的，该对象的类型决定了应该激活哪个catch的处理代码。由于有时候会有我们没有预测到的异常发生，不能判断是什么类型，所以我们一般会在函数中加一个任意类型的异常捕获，防止程序崩溃。
  3. 被选中的处理代码是调用链中与该对象类型匹配且离抛出异常位置最近的那一个。
  4. 异常的重新抛出：有可能单个的catch不能完全处理一个异常，在进行一些校正处理以后，希望再交给更外层的调用链函数来处理，catch则可以通过重新抛出将异常传递给更上层的函数进行处理。

异常的安全与规范

异常安全

  • 构造函数完成对象的构造和初始化，最好不要在构造函数中抛出异常，否则可能导致对象不完整或没有完全初始化。
  • 析构函数主要完成资源的清理，最好不要在析构函数内抛出异常，否则可能导致资源泄漏。
  • C++中异常经常会导致资源泄漏的问题，比如在new和delete中抛出了异常，导致内存泄漏，在lock和unlock之间抛出了异常导致死锁，通常通过智能指针来解决。

异常规范

  • 异常规格说明的目的是为了让函数使用者知道该函数可能抛出的异常有哪些。 可以在函数的后面接throw(类型)，列出这个函数可能抛掷的所有异常类型。
  • 函数的后面接throw()，表示函数不抛异常。
  • 若无异常接口声明，则此函数可以抛掷任何类型的异常。

throw()和noexcept的区别

从功能上来说，void func() throw(); 和 void func() noexcept; 的作用是相同的

区别：

  1. throw() 是在 C++11 之前存在的异常规范，称为 “dynamic exception specification”，它的意思是向编译器保证函数不会抛出异常。而 noexcept 是在 C++11 中引入的，称为 “no except specification”，它的意思是告诉编译器这个函数不会抛出异常。
  2. 表现形式不同。throw() 是放在函数声明和定义的末尾，如 void func() throw(); 和 void func() throw() { ... }。而 noexcept 是在函数后面，如 void func() noexcept; 和 void func() noexcept { ... }。
  3. 技术实现机制不同。throw() 机制会在函数调用前检查是否抛出异常，会消耗一定的性能，而 noexcept 不会在函数调用前做检查，它只是一个标志，不会产生额外的性能消耗。

因此，从性能和可读性方面来看，建议使用 noexcept 代替 throw()。但是需要注意的是，函数是否抛出异常，还需要根据具体实现进行分析和判断，不能单纯依赖于使用异常规范来控制。

基类和派生类的catch

派生类被throw，可以被基类catch，在派生类的 catch 之前应该先定义基类的 catch，否则基类的 catch 永远不会被执行。

noexcept

noexcept 关键字用来表明函数不会抛出异常，主要是帮助编译器进行优化以及在一些场景下能够更好地安全编程，比如在析构函数中使用 noexcept，确保在对象析构过程中不会抛出异常，从而避免内存泄漏等问题。

如果使用了 noexcept 关键字声明的函数在运行时抛出了异常，程序将会终止，即调用 std::terminate。通常，编译器会在运行时的最开始检查是否抛出了异常，并调用 std::terminate，所以在函数在抛出异常时，不会直接返回到调用它的地方。

这意味着，如果一个 noexcept 函数抛出了异常，程序可能会未经处理地终止，或调用 terminate 函数，这些都是无法回避或处理的。

因此，在编写程序时，应该谨慎使用 noexcept 关键字，只在确定不会抛出异常的情况下使用，同时需要对可能出现异常的代码进行防御性编程，避免程序异常终止带来的不可预知的后果。

new int（）是开辟一块int类型的内存，并初始化为括号里的值
new int [] 是开辟一块大小为中括号内个int大小的内存

delete 对于内置数据类型无论是否是数组，都可以不加[]，对于类类型的数组，则需要加中括号。因为不加[]只调用了下标为0的对象的析构函数，而加[]以后则会调用数组中每个对象的析构函数，再释放其内存

引用和原变量公用一块内存空间，不在开辟新空间
对于对象的引用必须是左值
sizeof中，引用的大小是其引用的对象的大小，这和指针不同，32位系统的指针大小是4字节也就是32位，64位系统是8字节也就是64位
