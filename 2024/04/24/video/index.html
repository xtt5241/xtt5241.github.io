视频

一、简介

视频技术

  • 封装技术
  • 视频压缩编码技术
  • 音频压缩编码技术
  • 流媒体协议技术（网络传输）

视频播放

在线视频：

  • 解协议
  • 解封装
  • 解码音视频
  • 音视频同步

本地视频：

  • 解封装
  • 解码音视频
  • 音视频同步

解协议的作用，就是将流媒体协议的数据，解析为标准的相应的封装格式数据

视音频在网络上传播的时候，常常采用各种流媒体协议，例如HTTP，RTMP，或是MMS等等。这些协议在传输视音频数据的同时，也会传输一些信令数据。这些信令数据包括对播放的控制（播放，暂停，停止），或者对网络状态的描述等。解协议的过程中会去除掉信令数据而只保留视音频数据。例如，采用RTMP协议传输的数据，经过解协议操作后，输出FLV格式的数据。

解封装的作用，就是将输入的封装格式的数据，分离成为音频流压缩编码数据和视频流压缩编码数据

封装格式种类很多，例如MP4，MKV，RMVB，TS，FLV，AVI等等，它的作用就是将已经压缩编码的视频数据和音频数据按照一定的格式放到一起。例如，FLV格式的数据，经过解封装操作后，输出H.264编码的视频码流和AAC编码的音频码流。

解码的作用，就是将视频/音频压缩编码数据，解码成为非压缩的视频/音频原始数据

音频的压缩编码标准包含AAC，MP3，AC-3等等，视频的压缩编码标准则包含H.264，MPEG2，VC-1等等。解码是整个系统中最重要也是最复杂的一个环节。通过解码，压缩编码的视频数据输出成为非压缩的颜色数据，例如YUV420P，RGB等等；压缩编码的音频数据输出成为非压缩的音频抽样数据，例如PCM数据。

视音频同步的作用，就是根据解封装模块处理过程中获取到的参数信息，同步解码出来的视频和音频数据，并将视频音频数据送至系统的显卡和声卡播放出来

视频格式

  • AVI（Audio Video Interleave）：是Microsoft开发的一种常见的视频容器格式，支持多种视频编解码器。
  • MP4（MPEG-4 Part 14）：是一种常见的多媒体容器格式，通常用于存储视频和音频流，支持多种编解码器和元数据。
  • MKV（Matroska Multimedia Container）：是一种开放的多媒体容器格式，支持多种视频、音频和字幕流，并且具有良好的扩展性和灵活性。
  • MOV（QuickTime File Format）：是由Apple开发的多媒体容器格式，常用于存储视频、音频和其他媒体数据，支持多种编解码器。
  • FLV（Flash Video Format）：是Adobe Flash Player使用的一种视频容器格式，通常用于网络视频流媒体。
  • WebM：是Google推出的开放媒体文件格式，主要用于网络视频，支持VP8、VP9等开放的视频编码格式。
  • H.264、H.265等编码格式：这些并非容器格式，而是视频编码标准，但在实际使用中，视频通常会以这些编码格式进行存储或传输。

视频流

  • RTSP（Real-Time Streaming Protocol）：是一种常见的实时流媒体传输协议，用于在网络上传输音视频流。
  • RTMP（Real-Time Messaging Protocol）：是Adobe开发的实时消息传输协议，常用于将音视频流传输到Flash播放器。
  • HLS（HTTP Live Streaming）：是由Apple开发的一种流媒体传输协议，通过HTTP协议在网络上分发媒体资源。
  • MPEG-DASH（Dynamic Adaptive Streaming over HTTP）：是一种动态自适应流媒体传输协议，通过HTTP协议提供多码率视频流。
  • SRT（Secure Reliable Transport）：是一种安全可靠的实时传输协议，适用于对网络质量有较高要求的音视频传输场景。

视频编码标准

视频编码标准通常由多个层组成，其中包括：

  1. Application Layer（应用层）NAL：应用层是视频编码标准的顶层，它定义了视频编码的应用场景和使用方式，例如视频会议、视频流媒体等。

  2. Video Coding Layer（视频编码层）VCL：视频编码层负责将原始视频数据转换为压缩格式。这一层包括视频编码算法和技术，如变换、运动估计、量化等，常见的视频编码标准如H.264、H.265等都包含了视频编码层。

  3. Network Abstraction Layer（网络抽象层）NAL：网络抽象层负责将视频编码层产生的数据进行打包和组织，以便在网络上传输。这一层定义了如何将视频数据打包成网络传输单元（如NAL单元），并添加必要的头部信息。

  4. Transport Layer（传输层）TL：传输层负责管理视频数据在网络中的传输，包括数据包的分发、丢失恢复、流量控制等。常见的传输协议有UDP（User Datagram Protocol）和TCP（Transmission Control Protocol）等。

  5. Physical Layer（物理层）PL：物理层是视频数据在网络传输中的最底层，它负责将数字数据转换为物理信号，并在网络中传输。这一层包括硬件设备和传输介质，如网线、光纤等。

这些层次通常通过标准化的协议和接口进行交互，以实现视频的编码、传输和解码。不同的视频编码标准可能在这些层次上有所不同，但通常都遵循类似的结构和原则。

H.264，也称为MPEG-4 Part 10或AVC（Advanced Video Coding，高级视频编码），是一种广泛使用的视频编码标准，用于压缩和传输数字视频。它被认为是一种高效的视频压缩标准，能够提供较高的视频质量和更低的比特率，适用于多种应用场景，包括广播、存储、视频会议、流媒体等。

  1. 高压缩性能：H.264采用了先进的编码技术，如运动补偿、帧内预测、变换编码等，能够在保持较高视频质量的前提下实现更高的压缩率，从而减少了视频数据的传输和存储成本。
  2. 多种帧类型：H.264定义了多种帧类型，包括关键帧（I帧）、预测帧（P帧）和双向预测帧（B帧），利用帧间的相关性和运动信息来进一步提高压缩效率。
  3. 灵活的码率控制：H.264支持灵活的码率控制机制，可以根据网络带宽和设备性能动态调整视频的码率，以适应不同的传输环境和应用需求。
  4. 网络友好性：H.264在设计时考虑了网络传输的特点，采用了网络抖动补偿、流量平滑和自适应编码等技术，提高了视频在不稳定网络环境下的传输稳定性和鲁棒性。
  5. 多分辨率支持：H.264支持多种视频分辨率和帧率，可以适应不同的显示设备和应用场景，从标清到高清甚至超高清，都有良好的支持。
  6. 广泛应用：H.264已成为许多应用中的标准编码格式，包括数字电视、视频会议、网络视频流媒体、蓝光光盘、移动视频通信等领域。

帧和RTP和NALU的关系是什么

帧（Frame）、RTP（Real-time Transport Protocol）和NALU（Network Abstraction Layer Unit）在视频传输中有着密切的关系：

  1. 帧（Frame）：

    • 在视频编码中，一幅图像被划分为一系列的帧。常见的帧类型包括关键帧（I帧）、预测帧（P帧）和双向预测帧（B帧）。关键帧是独立编码的，而P帧和B帧则是参考前后帧进行差异编码的。

  2. NALU（Network Abstraction Layer Unit）：

    • NALU 是视频编码器输出的最小单元，它将视频帧分解成一系列的数据单元。每个 NALU 包含了一个头部和相应的负载数据，用于在网络上传输视频数据。常见的 NALU 类型包括 SPS（Sequence Parameter Set）、PPS（Picture Parameter Set）、关键帧数据单元（IDR帧）和非关键帧数据单元（非IDR帧）等。

  3. RTP（Real-time Transport Protocol）：

    • RTP 是一种用于实时传输多媒体数据的协议，常用于音视频通信中。RTP 协议提供了序列号、时间戳等功能，用于保证数据传输的顺序和时序性，并且支持对多媒体数据进行分包、标识和传输。在实时视频传输中，视频数据通常被封装为 RTP 包进行传输。

这三者之间的关系如下：

  • 视频帧被编码成一系列的 NALU 单元。
  • 这些 NALU 单元通常被封装到 RTP 数据包中，以便通过网络进行传输。
  • RTP 包中包含了序列号、时间戳等信息，用于确保数据传输的时序性和可靠性。
  • 接收端接收到 RTP 包后，解析其中的 NALU 数据，进行解码和播放。

总的来说，帧被编码成NALU单元，然后封装到RTP包中进行传输，以实现实时视频传输的目的。

0. NALU

Network Abstraction Layer Unit——网络抽象层单元

NALU（Network Abstraction Layer Unit）是用于 H.264/AVC 和 H.265/HEVC 等视频编码标准中的基本数据单元。它是视频编码器输出的最小单元，用于在网络上传输视频数据。

NALU 是由一个字节头部（NALU Header）和一个负载（Payload）组成的。

而 Start Code 则是一种常见的用于标识 NALU 起始位置的机制，特别是在早期的视频编码标准中（如 H.264/AVC）。

在H.264流中，NALU按顺序传输，接收端通过解析NALU头部来识别和处理不同类型的NALU，以恢复出原始的视频数据。因此，NALU在H.264视频编码中起着非常重要的作用，定义了视频数据的组织和传输方式。

一个完整的 NALU 数据单元可能看起来像这样：

  1. 起始码（StartCode）：
    • 在 H.264/AVC 中，NALU 之间通常使用 Start Code 进行分隔。
      • Start Code 是一种特定的字节序列，用于标识一个 NALU 的开始位置。
      • 解码器在接收到视频数据流时，可以通过检测 Start Code 来识别每个 NALU 的开始位置，进而解析 NALU 的头部信息和负载数据。
    • Start Code 类型
      • 0x 00 00 00 01
      • 0x 00 00 01
  2. NALU 头部（NALU Header）：
    • NALU 头部通常是一个字节，其中包含了一些控制信息，用于描述 NALU 的类型和特性。头部的组成可以根据不同的视频编码标准而有所不同。通常情况下，头部的结构包含以下几个字段：
      • NALU 类型（NALU Type）：指示了这个 NALU 的类型，如关键帧、非关键帧、SPS（Sequence Parameter Set）、PPS（Picture Parameter Set）等。
      • 优先级（Priority）：指示了这个 NALU 的传输优先级。
      • 资源分配类型（Dependency ID）：指示了这个 NALU 的依赖关系，如其是否依赖于其他 NALU。
      • 一些保留位（Reserved Bits）：用于未来的标准扩展。
  3. NALU 负载（Payload）：
    • 负载部分是 NALU 的实际数据，其中包含了视频编码器输出的压缩视频数据。负载的具体内容取决于 NALU 的类型和编码标准，可以是压缩的视频帧数据、参数集等。

不同的 NALU 类型承载着不同类型的数据，例如：

  • 关键帧（I 帧）：包含完整的图像信息，可独立解码，通常用于视频序列的起始或关键位置。
  • 非关键帧（P 帧、B 帧等）：相对于关键帧，它们可能只包含图像的一部分，需要结合其他帧进行解码。
  • 参数集（SPS、PPS 等）：包含视频编码器的配置参数，如图像尺寸、帧率等，用于初始化解码器每个NALU之间使用，每个NALU的第一次字节都有特殊的含义

1. 两层功能

VCL（Video Coding Layer）

  • VCL是视频编码标准中的核心部分，负责对视频内容进行压缩和编码。
  • VCL层定义了视频编码的各种算法和技术，包括运动估计、变换编码、量化、熵编码等。
  • VCL层将原始的视频帧转换为编码后的数据流，以便传输或存储。

NAL（Network Abstraction Layer）

  • NAL是视频编码标准中的一个抽象层，用于封装和传输VCL产生的编码数据。
  • NAL层定义了将编码后的视频数据打包成网络传输所需的格式，包括分割成小的单元（NAL单元）、添加头信息等。
  • NAL单元是组成视频数据的最小单元，每个NAL单元包含了编码数据的类型和相关信息，以便接收端进行解码和处理。

两者关系如下

  • VCL层负责视频编码和解码的算法实现，将原始视频帧转换为编码数据或将编码数据解码为原始视频帧。
  • NAL层负责将VCL产生的编码数据进行封装和传输，使其适合在网络上进行传输或存储，同时提供了对编码数据的分割和解析机制。

2. 常用Nalu_Type：

  1. VCL NALU：
    • VCL NALU包含了视频编码的关键信息，如帧的类型（I帧、P帧、B帧等）和实际的像素数据。
    • 其中，I帧（Intra-coded picture）是一种完整的帧，不依赖于其他帧；P帧（Predictive-coded picture）和B帧（Bi-predictive-coded picture）是预测帧，依赖于其他帧进行解码。
    • VCL NALU的类型包括7、8、5、6等。
  2. 非VCL NALU：
    • 非VCL NALU包含了视频编码过程中的控制信息和辅助信息，如序列参数集（SPS，Sequence Parameter Set）、图像参数集（PPS，Picture Parameter Set）、片组（Slice Group）等。
    • 这些信息对视频解码器的配置和解码过程至关重要。

参数类型

序列参数集（SPS）

Sequence Paramater Set——序列参数集

SPS中保存了一组编码视频序列(Coded video sequence)的全局参数。所谓的编码视频序列即原始视频的一帧一帧的像素数据经过编码之后的结构组成的序列。而每一帧的编码后数据所依赖的参数保存于图像参数集中。

  • 视频编码层级（Profile & Level）：定义了使用哪些编码工具和约束，这影响到解码器必须支持哪些功能。
  • 图像尺寸和格式：包括宽度和高度，以及像素的宽高比。
  • 编码帧序列的方式：例如帧编码间隔、参考帧的数量等。
  • 时间尺度和帧率：定义视频的时间基准和帧的显示时间。
图像参数集（PPS）

Picture ParamaterSet——图像参数集

图像参数集（PPS）包含了影响单个图像或图像组（GOP，一组连续的帧）的参数，这些参数是解码器在解码具体一组图像时需要用到的。

  • 熵编码模式：指定用于编码切片数据的熵编码类型，例如CABAC（上下文自适应二进制算术编码）或CAVLC（上下文自适应变长编码）。
  • 切片组的配置：如何将图像分为多个切片，切片可以独立解码。
  • 权重预测参数：在预测帧时使用的加权因子，这对于场景转换时的平滑有重要作用。
  • 去块滤波的设置：去块滤波有助于减少压缩引起的块状伪影。
辅助增强信息（SEI）

Supplemental Enhancement Information——辅助增强信息

是H.264和其他视频编解码器中的一种元数据类型，用于在视频流中携带附加信息。SEI主要用于在视频传输和存储过程中提供额外的信息，以增强视频质量、提供额外的控制或者传递一些额外的信息。

  1. 视频参数集（VUI参数集）：提供有关视频时序信息（如宽高比、色度空间等）的附加信息。
  2. 图像截图：包含视频中某一时刻的截图图像，用于生成视频预览或者缩略图。
  3. 用户数据：允许开发者在视频流中插入自定义的数据，用于传递额外的信息，例如版权信息、广告等。
  4. 场景信息：描述视频中的特定场景，如场景切换、光线变化等。
  5. 错误恢复信息：提供视频帧丢失或损坏时的错误恢复策略。

SEI消息通常附加在视频码流中的某些特定位置，例如帧头、片头或GOP（Group of Pictures）头部。接收器在解码视频时可以解析这些SEI消息，并根据其中的信息进行相应的处理。

注意

一般情况SPS和PPS的NALU通常位于整个码流的起始位置。

但在某些特殊情况下，在码流中间也可能出现这两种结构，主要原因可能为：

  • 解码器需要在码流中间开始解码；
  • 编码器在编码的过程中改变了码流的参数（如图像分辨率等）；

在做视频播放器时，为了让后续的解码过程可以使用SPS中包含的参数，必须对其中的数据进行解析。

帧类型

I帧

（Intra-coded Picture）

  • 描述：I帧是关键帧，也称为帧内编码帧。
  • 特点：每个I帧都是独立编码的，不依赖于其他帧。它包含完整的图像信息，通常用于视频的开始或者关键场景。
  • 优点：提供了高质量的图像，但文件大小通常比其他类型的帧要大。
  • 用途：通常用作视频的开始或者在场景发生较大变化时使用。
P帧

（Predictive-coded Picture）

  • 描述：P帧是预测编码帧。
  • 特点：P帧依赖于之前的I帧或P帧进行编码，通过差异编码来表示图像的变化。
  • 优点：相比I帧，P帧的文件大小通常较小。
  • 用途：用于表示场景中的运动或者变化。
B帧

（Bidirectional-coded Picture）

  • 描述：B帧是双向预测编码帧。
  • 特点：B帧既依赖于之前的I帧或P帧，也依赖于之后的P帧或I帧。它可以表示两个参考帧之间的运动或变化。
  • 优点：通常可以进一步减小文件大小，因为它可以利用前后两个参考帧进行编码。
  • 用途：用于表示更复杂的运动或者变化。
IDR帧

（Instantaneous Decoding Refresh）

  • 描述：IDR帧是关键I帧。
  • 特点：IDR帧独立编码，不依赖于之前的任何帧。与普通的I帧相比，IDR帧更加关键，它会刷新解码器的参考图像。
  • 优点：提供了一个清晰的参考点，可以用于错误恢复或者随机访问。
  • 用途：通常用于关键的场景或者视频的开始，或者用于错误恢复。

3. H264码流进行RTP封装

首先要明确，RTP包的格式是绝不会变的，永远都是RTP头+RTP载荷

RTP头部是固定的，那么只能在RTP载荷中去添加额外信息来说明这个RTP包是表示同一个NALU

H264的三种RTP打包方式

  • 单NALU打包： 一个RTP包包含一个完整的NALU
  • 聚合打包：对于较小的NALU，一个RTP包可包含多个完整的NALU
  • 分片打包：对于较大的NALU，一个NALU可以分为多个RTP包发送

注意：这里要区分好概念，每一个RTP包都包含一个RTP头部和RTP荷载，这是固定的。而H.264发送数据可支持三种RTP打包方式

单NALU打包

所谓单NALU打包就是将一整个NALU的数据放入RTP包的载荷中，这是最简单的一种方式。

分片打包

RTP有MTU限制，MTU通常小于NALU的大小，所以需要分片打包，H264较常用的分片模式为FU-A

每个RTP包都有大小限制的，因为RTP一般都是使用UDP发送，UDP没有流量控制，所以要限制每一次发送的大小，所以如果一个NALU的太大，就需要分成多个RTP包发送，至于如何分成多个RTP包，如下：

RTP包结构

如果是分片打包的话，那么在RTP载荷开始有两个字节的信息（FU indicator和FU header），然后再是NALU的内容

FU-A分片方式

RTP头部之后的第一个字节为 FU indicator，第二个字节为 FU header，随后为FU payload

  • 第一个字节位FU Indicator

F：Nalu Header中对应的F

NRI：Nalu Header中对应的NRI

Type：分片类型，28 是特定用于指示 FU-A（Fragmentation Unit - Type A）的负载类型

  • 第二个字节位FU Header

S：start标记位，当为1时表示NALU分片的起始分片

E：end标记位，当为1时表示NALU分片的最后一个分片。

R：保留位，可忽略

Type：Nalu Header中对应的Type，等于帧类型

三个Type:

  • Nalu Header中的——Nal单元数据类型，也就是标识该NAL单元的数据类型是哪种，占用NAL头的第四到第8个位；

  • FU Indicator中的——分片类型，28 是特定用于指示 FU-A（Fragmentation Unit - Type A）的负载类型

  • FU Header中的——Nalu Header中对应的Type，等于帧类型

  • 随后为FU payload

NALU的数据，去掉起始位及NALU header之后的数据，按照分片大小进行分包

  • 注意

同一个帧的分片的开头两个字节除了起始和结束FU header中的S和E位不同，其他分片这两个字节都一样

这里起始分片要注意去掉Start Code 和Nalu Header，通过FU indicator的F/NRI以及FU header即可组合成NALU Header

RTP解封装的时候注意生成NALU头及起始标识。

一般H264进行RTP封装

  • SPS/PPS采用单一NALU封装方式
  • I帧/P帧采用FU-A分片模式
  • 如果带有SEI及AUD可过滤掉，也可以采用单一NALU封装方式

示例：

  • 0x7c:其二进制为：01111100，F=0，NRI=3，type=28表示FU-A分片，FU indicator

  • 0x85:其二进制为：10000101，S=1，E=0,type=5,表示I帧的起始FU-A分片，fu header
