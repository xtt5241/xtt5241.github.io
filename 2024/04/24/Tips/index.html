1.为什么要有头文件

偷懒用的

白话——相当于复制黏贴

本质是 每个源文件都是单独编译。头文件是为了确保单独编译成功，而为每个源文件提供名称声明。

  1. 声明接口: 头文件通常包含函数的原型声明、数据类型的定义、宏的定义等，它们定义了代码模块或库提供的接口。通过包含头文件，你可以在不同的源文件中使用这些接口。
  2. 避免重复: 如果同一声明或定义在多个源文件中重复出现，这将造成冗余和潜在的错误。头文件允许开发者在一个地方进行声明，然后在多个源文件中重用。
  3. 模块化: 头文件支持将程序分割成小的、可管理的、可重用的部分，这有助于组织代码，使其更加模块化。
  4. 隐藏实现细节: 通过在头文件中只提供接口声明，而将实现细节放在源文件中，可以隐藏实现的具体内容，这对于库的编写者尤其重要，因为他们可能不希望暴露内部实现。
  5. 提高编译效率: 如果没有头文件，每次编译程序时都必须重新处理所有的声明和定义。通过使用头文件，并结合编译器的预编译头功能或分离编译，可以减少编译时间。
  6. 促进团队合作: 在多人开发项目中，头文件可以使得不同的开发者在没有干扰到其他人的工作的情况下，独立地工作在不同的源文件上。
  7. 支持抽象和封装: 头文件提供了一种机制，允许开发者定义抽象数据类型和封装数据，这是面向对象编程的基本原则之一

2.头文件的#ifndef

#ifndef 是 C 和 C++ 编程语言中的一个预处理指令，常用于防止头文件的内容被多次包含。其作用和机制如下：

作用

#ifndef 指令用于检查某个宏是否未定义。如果未定义，随后的代码才会被编译器包含进来。这主要用于防止头文件被重复包含，从而避免重复定义同一变量或多次声明同一函数。

使用机制

  1. 防止头文件重复包含：当一个头文件可能被多个源文件包含时，头文件中的内容（如变量定义、函数声明）可能会被处理多次，导致编译错误。为了防止这种情况，通常会在头文件的开始使用 #ifndef，#define 和 #endif 来构建所谓的“包含卫士”（include guard）。

3.为什么头文件不能定义普通变量函数

在C或C++中，通常不建议在头文件中定义普通变量或函数（除非是声明为inline的函数），因为这样做可能会导致多重定义的问题。这里有几个主要原因：

多重定义问题

如果在头文件中定义了一个变量或函数，然后这个头文件被多个源文件包含，那么每个源文件都会得到一个该变量或函数的副本。这将导致链接错误，因为链接器会发现同一个符号有多个定义。

遵循“声明与定义分离”原则

通常的做法是在头文件中只声明变量和函数，而在源文件中进行定义。这样做的好处是：

  • 减少编译依赖：只需要包含变量或函数的声明，而不是整个定义，可以减少编译时间和重新编译的需要。
  • 提高代码的模块性：声明提供了一个接口，而定义则是实现。这样分离可以使得代码结构更清晰，也更容易管理。

防止链接器冲突

即使使用了包含卫士防止了头文件的多次包含，如果在头文件中定义了变量或非内联函数，同一个程序的不同编译单元（源文件）之间仍然可能因为包含了相同的头文件而导致定义冲突。

例外情况

  • 内联函数：可以在头文件中定义，因为内联函数在每个调用点上都被展开，不会导致多重定义的问题。
  • 常量：使用const（对于基础数据类型）或者constexpr（对于C++11及之后），可以在头文件中定义，因为它们具有内部链接性，不会导致链接错误。
  • 模板：模板的定义通常包含在头文件中，因为模板需要在编译时对每个使用的实例进行具体化。

总结来说，将变量和函数的定义放在源文件中，可以避免多重定义的问题，确保程序的链接正确无误，同时保持代码的结构清晰和模块化。

头文件的使用

推荐在main函数中包含多个头文件吗，还是说最好写多个自定义的头文件，在用到想用的其他头文件时，进行包含，然后main函数中直接包含这些自定的头文件

在C++项目中组织头文件和包含头文件的方式有很多种考虑因素，包括代码的可读性、维护性以及编译效率等。对于是否应该在 main.cpp 中直接包含多个头文件，或者通过自定义头文件来间接包含其他必需的头文件，以下是一些通用的建议：

  1. 最小包含原则：每个文件应该只包含它直接需要的头文件。这样可以减少编译时间，防止名称冲突和不必要的依赖。

  2. 自包含头文件：每个头文件应该足够独立，即包含它所需要的所有其他头文件。这样可以确保任何包含该头文件的源文件都能顺利编译。

  3. 避免头文件污染：通过在自定义头文件中合理地组织和包含其他头文件，可以避免在多个源文件中重复包含同一头文件，这有助于减少编译时间和避免潜在的编译问题。

  4. 使用预编译头文件：对于一些常用且不经常更改的头文件，可以使用预编译头文件来提高编译效率。

基于上述原则，你可以采取以下策略：

  • 如果 main.cpp 依赖于多个复杂的库或头文件，而这些依赖关系在项目中多次出现，可以创建一个或多个自定义头文件来管理这些依赖。例如，如果多个源文件都用到网络编程和数据库接口，可以创建 networking.h 和 database.h，在这些自定义头文件中包含必要的系统和第三方库头文件。

  • 在 main.cpp 或其他源文件中，只需包含这些高层自定义头文件，而不是直接包含所有单独的系统或库头文件。这样，源文件会更清洁，依赖关系更明确，同时还易于维护。

4.我在头文件中声明的函数，他在定义中所需的头文件，是包含在该头文件，还是源文件中

在C或C++编程中，函数在头文件中的声明和在源文件中的定义是分开的。通常，函数的声明（即函数原型）会放在一个头文件中（比如 .h 文件），而函数的定义（即函数的实际代码实现）则放在一个源文件中（比如 .cpp 或 .c 文件）。关于函数在定义中所需的头文件的包含问题，有以下两种常见的做法：

  1. 在源文件中包含所需的头文件：这是最常见的做法。如果函数的定义需要特定的库或者类型定义，通常会在这个函数的源文件中包含相应的头文件。这样做的好处是，只有实际使用到这些库的源文件才包含相关的头文件，可以避免不必要的编译依赖和编译时间。

  2. 在头文件中包含所需的头文件：有时，如果一个头文件中声明的函数或类型普遍依赖于某些其他头文件，可以直接在头文件中包含这些必需的头文件。这样做可以简化源文件的包含关系，但可能会导致较宽泛的编译依赖和潜在的编译效率问题，因为每个包含了该头文件的源文件都会间接包含更多的头文件。

总之，根据函数定义所依赖的具体内容来决定头文件的包含位置是比较灵活的。通常建议仅在需要时在源文件中包含必要的头文件，以减少不必要的依赖和提高编译效率。

变量命名

一、变量

  1. 局部变量
    1）定义循环：i,j,z,x,y;
    2）单个词时小写,多个词时，使用小驼峰命令：dataLength;
  2. 模块变量
    1）定义成static，使用匈牙利命名法+大驼峰命名法：s_DataLength;
  3. 全局变量
    1）使用匈牙利命名法+大驼峰命名法（帕斯卡命名法）：g_DataLength;
  4. 指针变量
    1）使用匈牙利命名法+大驼峰命名法：*p_DataLength;

二、函数

1）小驼峰命名法：fGetData(void);

三、宏定义

1）全部大写+下划线命名法：MAX_LENGTH

receiveLen还是recvLen

在变量命名的选择上，receiveLen 和 recvLen 都是可以接受的，但是它们各自的适用情景略有不同，这取决于代码的上下文以及预期的可读性和一致性。

  1. receiveLen:

    • 可读性: 这个命名更加清晰明了，receiveLen 完整地表达了变量的含义，即“接收长度”。这对于可能阅读或维护你代码的人来说，能更快地理解变量的用途。
    • 适用场景: 如果你的代码库中普遍采用完整的单词进行命名，或者你的代码将被不熟悉缩写的人阅读和使用，那么选择 receiveLen 更佳。

  2. recvLen:

    • 简洁性: recv 是 receive 的常见缩写，在许多编程环境中都很常见，特别是在网络编程中，如 socket 编程。因此，recvLen 提供了一种简洁且在某些领域内部广为接受的命名方式。
    • 适用场景: 如果你的代码主要是面向网络编程，或者团队中的成员都习惯于这种缩写，那么使用 recvLen 可能更合适。

推荐选择:

  • 如果你的目标是确保代码的清晰和易于理解，尤其是在一个团队环境中或当涉及到不同编程背景的开发者时，建议选择 receiveLen。
  • 如果你的代码环境中已经广泛使用了缩写，或者你需要与已有的代码库保持一致，那么使用 recvLen 可能更加合适。

总之，选择哪种命名应根据你的具体情况和团队的编码习惯来决定。同时，保持代码中命名的一致性也非常重要，这有助于提高代码的整体可读性和可维护性。

文本编辑器——回车和换行

当你在文本编辑器中按下回车键时，实际上发生了两个操作：回车（Carriage Return）和换行（Line Feed）。

  • 回车（Carriage Return）：它告诉打字机或文本编辑器将光标移动到当前行的开头，而不移动到下一行。
  • 换行（Line Feed）：它告诉打字机或文本编辑器将光标移动到下一行的开头。

这两个操作的组合在不同的操作系统中有不同的表示方式：

  • 在 Unix/Linux 系统中，换行符使用 \n 表示。
  • 在 Windows 系统中，换行符使用回车符 \r 和换行符 \n 的组合表示，即 \r\n。

因此，当你在文本文件中看到 \r\n 时，它表示将光标移到当前行的开头（回车），然后移动到下一行的开头（换行），这是在 Windows 环境下通常使用的行结束符。在 C 语言中，你经常会在字符串中看到这种表示，因为在处理文本数据时，需要考虑不同操作系统的行结束符。
