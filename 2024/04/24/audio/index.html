一、音频简介

经常见到这样的描述: 44100HZ 16bit stereo 或者 22050HZ 8bit mono 等等.

44100HZ 16bit stereo: 每秒钟有 44100 次采样, 采样数据用 16 位(2字节)记录, 双声道(立体声);

22050HZ 8bit mono: 每秒钟有 22050 次采样, 采样数据用 8 位(1字节)记录, 单声道;

当然也可以有 16bit 的单声道或 8bit 的立体声, 等等。

采样率是指：声音信号在“模→数”转换过程中单位时间内采样的次数。采样值是指每一次采样周期内声音模拟信号的积分值。

对于单声道声音文件，采样数据为八位的短整数（short int 00H-FFH）；

而对于双声道立体声声音文件，每次采样数据为一个16位的整数（int），高八位(左声道)和低八位(右声道)分别代表两个声道。

人对频率的识别范围是 20HZ - 20000HZ, 如果每秒钟能对声音做 20000 个采样, 回放时就足可以满足人耳的需求. 所以 22050 的采样频率是常用的, 44100已是CD音质, 超过48000的采样对人耳已经没有意义。这和电影的每秒 24 帧图片的道理差不多。

每个采样数据记录的是振幅, 采样精度取决于储存空间的大小:

1 字节(也就是8bit) 只能记录 256 个数, 也就是只能将振幅划分成 256 个等级;

2 字节(也就是16bit) 可以细到 65536 个数, 这已是 CD 标准了;

4 字节(也就是32bit) 能把振幅细分到 4294967296 个等级, 实在是没必要了.

如果是双声道(stereo), 采样就是双份的, 文件也差不多要大一倍.

这样我们就可以根据一个 wav 文件的大小、采样频率和采样大小估算出一个 wav 文件的播放长度。

譬如 “Windows XP 启动.wav” 的文件长度是 424,644 字节, 它是 “22050HZ / 16bit / 立体声” 格式(这可以从其 “属性->摘要” 里看到),

那么它的每秒的传输速率(位速, 也叫比特率、取样率)是 22050162 = 705600(bit/s), 换算成字节单位就是 705600/8 = 88200(字节/秒),
播放时间：424644(总字节数) / 88200(每秒字节数) ≈ 4.8145578(秒)。

但是这还不够精确, 包装标准的 PCM 格式的 WAVE 文件(.wav)中至少带有 42 个字节的头信息, 在计算播放时间时应该将其去掉,
所以就有：(424644-42) / (2205016*2/8) ≈ 4.8140816(秒). 这样就比较精确了.

关于声音文件还有一个概念: “位速”, 也有叫做比特率、取样率, 譬如上面文件的位速是 705.6kbps 或 705600bps, 其中的 b 是 bit, ps 是每秒的意思;

压缩的音频文件常常用位速来表示, 譬如达到 CD 音质的 MP3 是: 128kbps / 44100HZ.

三、AAC

简介

AAC（Advanced Audio Coding），中文名：高级音频编码。出现于1997年，基于MPEG-2的音频编码技术。由Fraunhofer IIS、杜比实验室、AT&T、索尼等公司共同开发，目的是取代MP3格式。

2000年，MPEG-4标准出现后，AAC重新集成了其特性，加入了SBR技术和PS技术，为了区别于传统的MPEG-2 AAC又称为MPEG-4 AAC。

本文简单地介绍AAC编码的一些基础概念，以及AAC格式的文件怎么组成的，需要了解更多AAC编码的内容，可以查看其相关文档《ISO-IEC 14496-3_2009》。

1. 两种音频格式：

  • ADIF：全称 Audio Data Interchange Format，音频数据交换格式，该格式一般应用在将音频通过写文件方式存储在磁盘里的场景，不能进行随机访问，不允许在文件中间开始进行解码；只有拿到整个文件时才能开始进行渲染播放；
  • ADTS：全称 Audio Data Transport Stream，音频数据传输流，该格式的特征是用同步字节进行将 AAC 音频截断，然后可以允许客户端在任何地方进行解码播放，适合网络传输场景；

简单来说，ADTS可以在任意帧进行解码，每一个帧都有头信息，一个帧就能单独解码。

但ADIF却只有一个统一的头，所以必须得到所有的数据后才能解码。

两种Header的形式也不同，目前一般编码后和抽取出的基本都是ADTS格式音频流。

syncword

syncword 是 ADTS 头部中的一个字段，用于同步和识别 AAC 数据流的起始点。具体介绍如下：

  • 类型：无符号整数
  • 位数：12位
  • 含义：syncword 是一个固定的 12 位字段，其取值为 1111 1111 1111，用于标识一个 ADTS 帧的开始。当解码器接收到这个同步字时，它就知道紧随其后的数据是 AAC 数据流，并且可以开始解析 ADTS 头部及其后续的音频数据。
  • 作用：通过这个同步字，解码器能够准确地找到每个 ADTS 帧的起始位置，从而正确地解析音频数据，实现音频的解码和播放。

在 AAC 数据流中，syncword 是一个关键的标识符，确保解码器能够正确地同步和解析音频数据，从而保证音频的连续播放和高质量解码。

id

ADTS 头部的 id 字段是一个 1 位 MPEG 标识符，用于指示 AAC 编码的 MPEG 标准版本。具体介绍如下：

  • 类型：无符号整数
  • 位数：1位
  • 含义：id 字段指示所使用的 AAC 编码的 MPEG 标准版本。通常，它被设置为 0 表示 MPEG-4，或者 1 表示 MPEG-2。MPEG-4 AAC 是更先进、更高效的编码标准，而 MPEG-2 AAC 则是更早期的标准。
  • 作用：id 字段使解码器能够识别所使用的 AAC 编码的 MPEG 标准版本，以便正确地解码音频数据。根据标准的不同，解码器可能会采用不同的解码算法和参数配置，以确保对音频数据的正确解析和播放。

通过 id 字段，解码器可以在解析 ADTS 头部时确定所使用的 AAC 编码的 MPEG 标准版本，并相应地进行解码处理，以实现音频数据的有效解析和播放。

layer

实际上，ADTS 头部确实包含了一个名为 “layer” 的字段，但它始终被设置为 “00”。这个字段通常被称为 “MPEG Layer” 字段，虽然在 AAC 中没有实际的层级，但是为了与 MPEG-1 标准保持一致，这个字段仍然存在于 ADTS 头部中，并且始终被设置为 “00”。

这个字段的存在主要是为了与其他 MPEG 标准兼容，并且在 AAC 中没有实际的用途。所以你在看到 ADTS 头部中的 “layer” 字段时，可以忽略它的值，因为它始终为 “00”。

protectionAbsent

ADTS头部的protectionAbsent字段是一个单比特的标志位，用于指示ADTS帧是否启用了CRC（Cyclic Redundancy Check）错误检测和纠正机制。

  • 类型： 单比特标志位
  • 含义： protectionAbsent字段取值为0时，表示ADTS帧启用了CRC错误检测和纠正机制；取值为1时，表示ADTS帧未启用CRC错误检测和纠正机制。
  • 作用： protectionAbsent字段的目的在于告知解码器在解析ADTS帧时是否需要执行CRC检测和纠正。启用CRC机制能够增强音频数据的可靠性，使得解码器能够在传输过程中检测和纠正可能发生的错误，从而减少音频数据的丢失或损坏。

通过protectionAbsent字段，解码器可以确定是否需要执行CRC错误检测和纠正，以确保对音频数据的正确解析和处理。

profile

在ADTS头部中，profile字段用于指示AAC音频的编码配置文件。这个profile字段是两位的，其中第一个位表示主要配置（Main Profile），第二个位表示扩展配置（Extended Profile）。

这两位的组合可以表示不同的AAC编码配置文件，从而支持不同的音频编码特性和性能。AAC编码器可以选择不同的配置文件来平衡音频质量、压缩效率和设备兼容性。下面是一些常见的AAC配置文件：

  1. Main Profile (00)：主要配置文件，提供了基本的AAC编码特性，适用于大多数应用场景。
  2. LC (Low Complexity) Profile (01)：低复杂度配置文件，相比于Main Profile，具有更低的计算复杂度和更高的压缩效率，适合于低功耗设备和带宽受限环境。
  3. SSR (Scalable Sample Rate) Profile (10)：可扩展采样率配置文件，支持不同的采样率，适用于需要动态调整采样率的场景。
  4. LTP (Long Term Prediction) Profile (11)：长期预测配置文件，通过长期预测技术提高了编码效率，适合于高质量音频的编码。
  5. LD (Low Delay) Profile：低延迟配置文件，用于对实时性要求较高的应用，如音视频通话等。

其他配置文件还包括了针对特定应用场景或者特殊需求的配置，这些配置文件都由对应的profile字段来表示。通过这种方式，AAC可以提供灵活性和扩展性，以满足不同应用场景下的需求。

AAC共有9种规格(Profile)，以适应不同的场合的需要：

  • MPEG-2 AAC LC：低复杂度规格（Low Complexity） 注：比较简单，没有增益控制，但提高了编码效率，在中等码率的编码效率以及音质方面，都能找到平衡点。
  • MPEG-2 AAC Main：主规格
  • MPEG-2 AAC SSR：可变采样率规格（Scaleable Sample Rate）
  • MPEG-4 AAC LC：低复杂度规格（Low Complexity）—现在的手机比较常见的MP4文件中的音频部份就包括了该规格音频文件
  • MPEG-4 AAC Main：主规格 注：包含了除增益控制之外的全部功能，其音质最好
  • MPEG-4 AAC SSR：可变采样率规格（Scaleable Sample Rate）
  • MPEG-4 AAC LTP：长时期预测规格（Long Term Predicition）
  • MPEG-4 AAC LD：低延迟规格（Low Delay）
  • MPEG-4 AAC HE：高效率规格（High Efficiency）—这种规格适合用于低码率编码，有Nero ACC 编码器支持

privateBit

ADTS 头部中的 privateBit

  • 类型：单个比特
  • 位数：1 位
  • 含义：privateBit 是一个单个比特，用于 AAC 数据流中的私有用途。它提供了额外的灵活性，允许编码器或解码器在音频数据流中添加自定义的私有信息或标志。通常情况下，这个比特位由编码器设置，并且可以被解码器用于特定的目的。
  • 作用：privateBit 的作用在于允许音频编码器和解码器在需要时传递额外的信息，例如特定的设置、标志或其他私有数据。这使得 AAC 数据流可以承载更多的信息，以满足特定应用或设备的需求。

channel_configuration

ADTS（音频数据传输流）头部中的 channel_configuration 字段用于指示音频通道的配置信息，即描述了音频数据流中声道的数量和布局。以下是关于 channel_configuration 字段的详细介绍：

  • 类型：无符号整数
  • 位数：4位
  • 含义：channel_configuration 字段指定了音频数据流中的声道配置。它表示了编码器使用的声道数以及声道的布局方式。具体取值可以是 0 到 7，其中：
    • 0：单声道（mono）配置，表示音频数据流只包含一个声道。
    • 1：双声道（stereo）配置，表示音频数据流包含两个声道，左声道和右声道。
    • 2-7：多声道配置，表示音频数据流包含更多的声道，具体的声道布局方式取决于具体的数值。例如：
      • 2：双声道配置（与值 1 相同）。
      • 3：3 声道配置，通常是左前、右前和中心。
      • 4：4 声道配置，通常是左前、右前、左后和右后。
      • 5：5 声道配置，通常是左前、右前、中心、左后和右后。
      • 6：5 声道配置，通常是左前、右前、中心、左侧和右侧。
      • 7：7 声道配置，通常是左前、右前、中心、左侧、右侧、左后和右后。
  • 作用：channel_configuration 字段的作用在于告知解码器如何解析音频数据流中的声道信息。通过正确解析声道配置，解码器可以将音频数据正确地路由到相应的声道输出，实现适当的声场效果和立体声感觉。

channel_configuration 字段在 ADTS 头部中扮演着关键的角色，确保解码器能够正确地理解音频数据流中声道的配置信息，并根据配置信息进行声音的解码和播放。

orininal_copy

ADTS 头部中的 original_copy 字段是一个单个比特位，用于指示编码的音频数据是否为原始副本（original）还是一个副本（copy）。这个字段的具体含义如下：

  • 类型：单个比特

  • 位数：1 位

  • 含义：original_copy 字段用于标识编码的音频数据是否为原始副本。

    • 如果 original_copy 的值为 0，则表示编码的音频数据是原始数据，即未经过任何副本处理。
    • 如果 original_copy 的值为 1，则表示编码的音频数据是一个副本，即它是对原始音频数据的复制或转码。

  • 作用：该字段的作用在于向解码器传递关于音频数据原始性的信息。解码器可以根据这个字段来做出不同的处理，例如对原始数据进行特定的音频处理或保留音频数据的完整性。

在音频传输和处理中，original_copy 字段可以帮助解码器根据需要进行适当的音频处理，从而提供更好的音频质量和用户体验。

home

编码时设置为0，解码时忽略。

copyrigth_identification_bit

ADTS 头部的 copyright_identification_bit 字段是一个 1 位的无符号整数，用于指示 AAC 编码的音频数据是否受到版权保护。

  • 类型：无符号整数
  • 位数：1位
  • 含义：copyright_identification_bit 字段指示音频数据是否受到版权保护。当该位被设置为 1 时，表示音频数据受到版权保护；当该位被设置为 0 时，表示音频数据不受版权保护。
  • 作用：该字段使解码器能够识别音频数据是否受到版权保护，以便在必要时采取相应的授权和处理措施。根据该位的设置，解码器可以决定是否允许对音频数据进行复制、分发或其他操作，以确保音频数据的合规性和版权保护。

通过 copyright_identification_bit 字段，解码器可以在解析 ADTS 头部时确定音频数据的版权保护状态，并相应地采取必要的措施，以保护版权并确保音频数据的合法使用。

copyrigth_identification_stat

ADTS 头部的 copyright_identification_stat 字段是一个 1 位的无符号整数，用于指示 AAC 编码的音频数据是否受到版权保护。

  • 类型：无符号整数
  • 位数：1位
  • 含义：copyright_identification_stat 字段指示音频数据是否受到版权保护。当该位被设置为 1 时，表示音频数据受到版权保护；当该位被设置为 0 时，表示音频数据不受版权保护。
  • 作用：该字段使解码器能够识别音频数据是否受到版权保护，以便在必要时采取相应的授权和处理措施。根据该位的设置，解码器可以决定是否允许对音频数据进行复制、分发或其他操作，以确保音频数据的合规性和版权保护。

通过 copyright_identification_stat 字段，解码器可以在解析 ADTS 头部时确定音频数据的版权保护状态，并相应地采取必要的措施，以保护版权并确保音频数据的合法使用。

aac_frame_length

ADTS 头部的 aac_frame_length 字段是一个 13 位的无符号整数，用于指示 AAC 音频帧的长度。

  • 类型：无符号整数
  • 位数：13位
  • 含义：aac_frame_length 字段表示 AAC 音频帧的长度，即音频数据的帧大小（以比特为单位）。这个字段包括了 ADTS 头部的长度以及音频帧的有效数据长度。AAC 音频帧的长度可以根据这个字段来确定，以便解码器正确地解析音频数据。
  • 作用：该字段使解码器能够识别 AAC 音频帧的长度，以便正确地提取和解码音频数据。通过这个字段，解码器可以知道每个音频帧的大小，并相应地进行解码处理，以实现音频数据的有效解析和播放。

通过 aac_frame_length 字段，解码器可以在解析 ADTS 头部时确定 AAC 音频帧的长度，并相应地进行解码处理，以确保音频数据的正确解析和播放。

adts_bufferfullness

ADTS 头部的 adts_buffer_fullness 字段是一个 11 位的无符号整数，用于指示解码器输入缓冲区的填充状态。

  • 类型：无符号整数
  • 位数：11位
  • 含义：adts_buffer_fullness 字段表示解码器输入缓冲区的填充状态，即缓冲区中已经存储的音频数据的比特数。它指示了解码器当前接收到的音频数据量，以便解码器能够适时地处理音频数据，并确保解码过程的平稳进行。
  • 作用：该字段使解码器能够了解输入缓冲区的填充状态，以便在解码过程中进行合适的数据处理和管理。通过监测 adts_buffer_fullness 字段，解码器可以根据缓冲区的填充情况来调整解码速度或采取其他措施，以避免缓冲区溢出或欠流，从而确保音频数据的连续解析和播放。

通过 adts_buffer_fullness 字段，解码器可以实时监测解码器输入缓冲区的填充状态，以保证音频数据的有效解析和流畅播放。

number_of_raw_data_blocks_in_frame

ADTS 头部的 number_of_raw_data_blocks_in_frame 字段是一个 2 位的无符号整数，用于指示每个 AAC 帧中包含的原始数据块（Raw Data Block）的数量。

  • 类型：无符号整数
  • 位数：2位
  • 含义：number_of_raw_data_blocks_in_frame 字段表示每个 AAC 帧中包含的原始数据块的数量。AAC 原始数据块是音频编码中的基本单元，用于表示音频信号的一部分。这个字段告诉解码器每个帧中有多少个原始数据块，以便正确解析音频数据。
  • 取值范围：由于是 2 位字段，可能的取值范围为 0 到 3。不同的取值表示每个帧中包含的原始数据块的数量不同。
    • number_of_raw_data_blocks_in_frame 的取值与 AAC 音频的编码配置和帧结构有关，不同的取值对应不同的帧结构和数据块组织方式。
  • 作用：解码器根据 number_of_raw_data_blocks_in_frame 字段确定每个 AAC 帧中原始数据块的数量，以便正确地解析和处理音频数据。该字段在解码过程中用于帮助解码器分割和处理音频数据，以恢复原始音频信号。

通过 number_of_raw_data_blocks_in_frame 字段，解码器可以了解每个 AAC 帧的内部结构和组织方式，从而有效地进行解码和音频数据处理。
