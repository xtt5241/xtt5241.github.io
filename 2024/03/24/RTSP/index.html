RTSP协议族

三大协议

RTSP是一个实时传输流协议，是一个应用层的协议。通常说的RTSP 包括RTSP协议、RTP协议、RTCP协议，对于这些协议的作用简单的理解如下

RTSP协议：

负责服务器与客户端之间的请求与响应

Real-Time Streaming Protocol——实时流传输协议

  • RTSP是用于控制流媒体服务器的协议，允许客户端控制媒体会话，如播放、暂停、停止等。
  • 客户端通过RTSP与服务器建立连接，并发送控制命令来操作媒体会话。
  • RTSP并不传输媒体数据，而是控制媒体会话的行为。

RTP协议：

负责服务器与客户端之间传输媒体数据

Real-time Transport Protocol——实时传输协议

  • RTP是一种用于在网络上传输实时数据的协议，如音频和视频流。
  • 它提供了时间戳、序列号和负载类型等信息，以便于在接收端重构出连续的媒体数据。
  • RTP并不关心数据如何传输，它只负责在传输过程中添加一些元数据，以便于接收端进行媒体数据的恢复和播放。

RTCP协议：

负责提供有关RTP传输质量的反馈，就是确保RTP传输的质量

RTP Control Protocol——RTP控制协议

  • RTCP是RTP的配套控制协议，用于提供会话控制和质量反馈。
  • RTCP负责周期性地向参与RTP会话的所有成员发送控制信息，包括参与者的网络状态、接收质量报告等。
  • RTCP信息的反馈可以被用于调整发送速率、质量控制等操作，以提高流媒体传输的效率和质量。

三者的关系：

  • RTSP并不会发送媒体数据，只是完成服务器和客户端之间的信令交互
  • RTP协议负责媒体数据传输，RTCP负责RTP数据包的监视和反馈。
  • RTP和RTCP并没有规定传输层的类型，可以选择udp和tcp。RTSP的传输层则要求是基于TCP。

二、RTP

RTP头的结构体

CSRC

CSRCContributing Source标识符用于标识贡献源Contributing Sources

CSRC列表是一个可变长度的字段，用于指示参与当前RTP数据包的所有源。每个CSRC标识符都代表一个贡献源，通常是参与混合、合成或产生当前RTP数据包的源。这些贡献源可以是音频混合器中的不同参与者或者是参与多点会议的不同成员。

CSRC列表的存在使得接收者能够识别出哪些源对当前数据包做出了贡献，以便在接收端进行适当的处理，比如恢复丢失的数据、实现混音等操作。

每个CSRC标识符通常由32位组成，用于唯一地标识参与当前数据包的每个源。

在RTP头部中，CSRC列表的长度由RTP头部中的CSRC计数器字段指定，以便接收者知道CSRC列表的长度，并能够正确解析每个CSRC标识符。

Padding

  • Padding字段用于指示RTP数据包中是否存在填充字节。填充字节用于确保RTP数据包达到指定的长度，通常是为了满足某些传输协议对数据包长度的要求。
  • 如果Padding字段的值为1，则表明RTP数据包的末尾包含填充字节
  • 接收端在解析数据包时需要忽略这些填充字节。
  • 填充字节的内容可以是任意值，但接收端必须正确处理填充字节，以确保接收到的数据包内容是正确的。

Extension

  • Extension字段用于指示RTP数据包中是否包含扩展头部。扩展头部用于在RTP数据包中添加额外的信息，如时间戳、同步信号、参考信息等。
  • 如果Extension字段的值为1，则表明RTP数据包中包含扩展头部。
  • 扩展头部的具体格式由RTP头部中的Extension字段后面的一些字段定义，包括扩展头部的长度和扩展头部的内容。
  • 扩展头部的内容是可选的，可以根据需要进行定义和使用。接收端在解析数据包时需要根据扩展头部的格式来正确地解析和处理其中的信息。

Version

  • 该字段是一个2位的无符号整数，用于标识RTP协议的版本。当前版本的RTP协议是RTP版本2（RTPv2），其Version字段的值为2。
  • RTP的版本号可以用来识别和区分不同版本的协议，并确保接收端能够正确地解析和处理收到的RTP数据包。如果接收端检测到不支持的RTP版本号，通常会丢弃该数据包或者发送相应的错误消息。

Payload

  • Payload Type 字段用于指示 RTP 数据包中的有效载荷的类型，即数据的编码格式或者数据流的类型。
  • 例如，它可以指示音频数据的编码格式（如PCM、AAC、Opus等）或视频数据的编码格式（如H.264、VP9等）。
  • Payload Type 的范围是 7 位，可以表示 0 至 127 之间的整数。不同的 Payload Type 值对应着不同的数据类型或编码格式，这些值可以在 RFC 文档中找到相应的定义或约定。

Marker

  • Marker 字段是一个单独的比特位，用于向接收端传递额外的控制信息。
  • 它可以用来标记 RTP 数据流中的重要事件或特殊帧。
  • Marker 位的具体含义由应用程序或协议规范定义。
  • 例如，在视频传输中，Marker 位可能用于标记关键帧（I-frame）或其他重要的帧，以便接收端能够正确地处理数据流。

Seq

Sequence Number（序列号）

  1. 数据包排序：Sequence Number字段用于帮助接收端对RTP数据包进行排序。由于RTP数据包在网络传输过程中可能会出现乱序、丢失或延迟的情况，因此接收端可以通过检查数据包的序列号来重新排序数据包，以便按正确的顺序处理它们。
  2. 数据包丢失检测：接收端可以通过检查序列号来检测数据包是否丢失。如果接收端发现接收到的数据包的序列号与预期的序列号不匹配，就可以判断在传输过程中可能发生了数据包丢失。
  3. 同步：在多媒体流传输中，接收端可以利用序列号来保持与发送端的同步。通过跟踪序列号的变化，接收端可以准确地了解数据包的传输顺序和时序。

Sequence Number字段的初始值通常由发送端随机选择，以避免潜在的冲突。每发送一个RTP数据包，发送端会将序列号增加1。因此，接收端可以通过检测序列号的递增来确认数据包的顺序。

TimeStamp

Timestamp（时间戳）

  1. 时序同步：Timestamp字段允许接收端根据发送端的时钟信息来同步媒体流的播放。通过将采样时刻映射到时间戳，接收端可以根据时间戳来恢复发送端的采样时序，并确保媒体流的正确播放顺序。
  2. 时延计算：接收端可以利用时间戳来计算媒体流在传输过程中的时延。通过比较不同数据包的时间戳，接收端可以估算出数据包在网络中传输所经历的时间，并计算出媒体流的延迟情况。
  3. 时钟频率信息：Timestamp字段通常与RTP头部中的时钟频率信息一起使用，以确定时间戳的单位。时钟频率信息通常在RTP会话开始时通过SDP（Session Description Protocol）协商确定，用于将时间戳值映射到实际的时间单位（如毫秒或微秒）。

在实时音视频传输中，Timestamp字段对于确保媒体流的同步和时序非常重要。通过将采样时刻映射到时间戳，并利用时钟频率信息，接收端可以准确地恢复出发送端的采样时序，并实现同步的播放效果。

SSRC

SSRC（同步源标识符）是RTP协议（实时传输协议）头部的一个重要组成部分。它是一个32位的标识符，用于标识RTP数据包的源。在RTP通信过程中，SSRC的主要作用是识别各种同步源，以区分不同的媒体流。

SSRC的功能和重要性
  1. 流识别：SSRC确保在多流传输过程中，接收方可以区分来自不同源的数据包。例如，在一个视频会议中，可能有多个参与者同时发送音频和视频流，每个流都会有一个唯一的SSRC，接收方可以据此区分各个流。
  2. 冲突管理：在RTP会话中，如果发现SSRC冲突（即两个源偶然使用了相同的SSRC），参与者可以选择一个新的SSRC，以解决冲突。
  3. 会话管理和控制：在RTP控制协议（RTCP）中，SSRC用于关联媒体流与其控制信息，帮助执行如流量控制和监测等任务。
SSRC的生成

SSRC应当是随机生成的，以确保在广泛的网络环境中的唯一性和随机性。在RTP会话开始时，发送方生成一个随机的SSRC值。如果在会话中检测到SSRC的冲突，发送方需要生成一个新的SSRC值。

SSRC的设计确保了RTP能够支持复杂的多媒体会议、广播和其他实时通信应用，同时提供必要的流控制和同步功能。

四、其他协议

1. SDP

Session Description Protocol——会话描述协议

一种网络协议，用于描述多媒体通信会话的信息。

大白话——用文本描述的通信各端（PC 端、Mac 端、Android 端、iOS 端等）的能力。

这种协议不传输媒体本身，而是用于协商网络中两个或多个参与者之间的会话设置。SDP通常用于互联网电话（VoIP）、视频会议等应用中，它可以定义参与会话的媒体类型、数据传输协议、媒体流端口以及其他必要的配置信息。

SDP广泛应用在实时传输协议（RTP）中，特别是在实现语音和视频通话的应用中，比如Skype或者WebRTC技术。SDP通过在通信前提供必要的会话信息，使得参与者能够正确连接并交换媒体流。

关系

SDP和RTP的关系在于：

  • SDP通常用于描述会话中包含的媒体流的信息，包括媒体类型、编码格式、传输地址等。
  • RTP则负责实际传输媒体数据，但它需要依赖SDP提供的描述信息来确定媒体数据的类型、编码格式等参数，以便于正确解析和处理媒体数据。

SDP包括三个组件：

1、网关（Gateway）：负责保护业务系统，防护各类网络攻击，只允许来自合法客户端的流量通过；

2、客户端（Client）：负责验证用户身份，把访问请求转发给网关；

3、管控端（Controller）：负责身份认证、配置策略，管控整个过程。

SDP的5重防御机制：

（1）SPA（单包授权认证）：网关只有接收到了客户端发出的SPA包，验证合法之后，才对该客户端的IP开放指定端口。这个技术可以屏蔽绝大多数非法用户的网络攻击，让漏洞扫描、DDoS等攻击方式都失效。

（2）MTLS（双向认证）：网关跟客户端的通信是加密的，而且是双向认证，网关认证用户，用户也要认证网关。双向的认证保证了中间人攻击无法奏效。

（3）动态防火墙：经过SPA认证后，网关会放行指定端口。但端口放行是暂时的，几秒内没有操作，端口就会自动关闭。最大限度提高防护强度。

（4）设备验证：SDP不止验证用户身份，还要验证用户设备。设备的验证包括设备健康状态的验证，如是否装了杀毒软件等等。设备验证还包括设备证书的验证，只有合法设备才会安装证书，证书参与通信数据的加密过程。保证连接都是来自合法设备的。

（5）应用绑定：用户只能访问有授权的应用。这符合“最小化授权”原则，保证了威胁无法横向蔓延。

常见的DDoS攻击有三种：

（1）http攻击：与服务器建立大量http连接，让服务器耗尽精力，无力服务正常用户。

（2）Tcp syn攻击：通过向服务器发送大量syn包，让服务器始终处于等待状态。

（3）udp反射攻击：伪造udp请求发给第三方dns或ntp服务器，把udp包中的发起者改为攻击目标的IP，让第三方服务器反射并放大自己的攻击，对攻击目标造成大量冲击。

SDP针对三种攻击的防护机制分别是：

（1）http攻击：SDP根本不与非法用户建立http连接。

（2）Tcp syn攻击：SDP会把所有非法syn包直接丢掉，不会等待。

（3）udp反射攻击：SDP可以对一些非公开的dns或ntp进行保护，避免恶意udp包触达这些服务。

SDP适用场景总结

SDP架构需要用户安装客户端，所以SDP对终端的安全控制更强，客户端和网关联动能产生更严密的防护效果。但客户端也限制了SDP的应用场景。一些公开的网站没法用SDP保护。

SDP特别适用于合作伙伴、供应商、第三方人员、分支机构等等特定人群访问业务系统的场景。这些系统需要在互联网上开放，以便第三方人员访问。但是又不需要向全世界所有人开放，引来黑客攻击。这种场景下，SDP能保证合法用户正常连接，对未知用户“隐形”。SDP在这类场景下是相当能打的。
