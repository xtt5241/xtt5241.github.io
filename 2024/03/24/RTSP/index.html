RTSP协议族

三大协议

RTSP是一个实时传输流协议，是一个应用层的协议。通常说的RTSP 包括RTSP协议、RTP协议、RTCP协议，对于这些协议的作用简单的理解如下

RTSP协议：

负责服务器与客户端之间的请求与响应

Real-Time Streaming Protocol——实时流传输协议

  • RTSP是用于控制流媒体服务器的协议，允许客户端控制媒体会话，如播放、暂停、停止等。
  • 客户端通过RTSP与服务器建立连接，并发送控制命令来操作媒体会话。
  • RTSP并不传输媒体数据，而是控制媒体会话的行为。

RTP协议：

负责服务器与客户端之间传输媒体数据

Real-time Transport Protocol——实时传输协议

  • RTP是一种用于在网络上传输实时数据的协议，如音频和视频流。
  • 它提供了时间戳、序列号和负载类型等信息，以便于在接收端重构出连续的媒体数据。
  • RTP并不关心数据如何传输，它只负责在传输过程中添加一些元数据，以便于接收端进行媒体数据的恢复和播放。

RTCP协议：

负责提供有关RTP传输质量的反馈，就是确保RTP传输的质量

RTP Control Protocol——RTP控制协议

  • RTCP是RTP的配套控制协议，用于提供会话控制和质量反馈。
  • RTCP负责周期性地向参与RTP会话的所有成员发送控制信息，包括参与者的网络状态、接收质量报告等。
  • RTCP信息的反馈可以被用于调整发送速率、质量控制等操作，以提高流媒体传输的效率和质量。

三者的关系：

  • RTSP并不会发送媒体数据，只是完成服务器和客户端之间的信令交互
  • RTP协议负责媒体数据传输，RTCP负责RTP数据包的监视和反馈。
  • RTP和RTCP并没有规定传输层的类型，可以选择udp和tcp。RTSP的传输层则要求是基于TCP。

一、RTSP

一、RTSP协议概述

RTSP(Real-TimeStream Protocol )是一种基于文本的应用层协议，在语法及一些消息参数等方面，RTSP协议与HTTP协议类似。

RTSP被用于建立的控制媒体流的传输，它为多媒体服务扮演“网络远程控制”的角色。尽管有时可以把RTSP控制信息和媒体数据流交织在一起传送，但一般情况RTSP本身并不用于转送媒体流数据。媒体数据的传送可通过RTP/RTCP等协议来完成。

二、基本的RTSP操作过程是:

  1. 建立连接：客户端向服务器发送连接请求，通常使用 TCP 进行连接。RTSP 默认端口是 554。

  2. 描述会话：客户端发送 DESCRIBE 请求，请求服务器提供关于媒体流的描述信息，包括媒体类型、编码格式、时长等。

  3. 建立会话：客户端发送 SETUP 请求，用于建立媒体流的传输通道。在这一步中，客户端和服务器协商传输协议（如 RTP/UDP 或 RTP/TCP）、传输端口等参数。

  4. 播放或暂停：客户端发送 PLAY 或 PAUSE 请求，用于控制媒体流的播放或暂停。

  5. 获取媒体数据：一旦会话建立成功，服务器开始向客户端发送媒体数据。客户端通过接收 RTP数据包来获取音视频数据。

  6. 关闭连接：当播放完成或用户结束会话时，客户端发送 TEARDOWN 请求，用于关闭 RTSP 会话和释放资源。

这些步骤构成了基本的 RTSP 操作过程。在实际应用中，还可以使用其他扩展功能，如设置流的速率、跳转到特定的时间点等。

三、RTSP协议与HTTP协议区别

  1. 用途：
    • HTTP 主要用于传输超文本（如网页、图片等）和其他静态资源，通常用于网页浏览和下载文件。
    • RTSP 主要用于控制实时流媒体的传输，如音视频流，通常用于视频会议、网络直播等场景。
  2. 传输方式：
    • HTTP 使用请求-响应模型，客户端发送请求，服务器返回响应，通常基于 TCP 进行传输。
    • RTSP 也使用请求-响应模型，但其设计用于流媒体控制，可以控制播放、暂停、跳转等操作，通常也基于 TCP 进行传输。
  3. 会话控制：
    • HTTP 是无状态协议，每个请求-响应都是相互独立的，服务器不会保存客户端的状态信息。
    • RTSP 是有状态协议，通过建立会话来管理客户端和服务器之间的状态，可以在会话中保持客户端的播放位置、参数设置等信息。
  4. 支持的数据类型：
    • HTTP 主要用于传输各种类型的数据，包括文本、图像、视频等。
    • RTSP 主要用于控制实时流媒体的传输，如音频和视频流。
  5. 协议扩展：
    • HTTP 支持多种扩展，如 HTTPS、HTTP/2、WebSocket 等，用于提供更安全、更高效的网络通信。
    • RTSP 也支持一些扩展功能，如 RTSP over HTTP、RTSP over WebSocket 等，用于在不同环境下提供更灵活的流媒体控制。
  6. 请求：
    • RTSP协议的客户端和服务器端都可以发送Request请求
    • HTTPF协议，只有客户端能发送Request请求。
  7. 载荷数据传输方式：
    • RTSP协议中，载荷数据一般是通过带外方式来传送的(除了交织的情况)，即通过RTP协议在不同的通道中（RTP、RTCP）来传送载荷数据。
    • HTTP协议的载荷数据都是通过带内方式传送的，比如请求的网页数据是在回应的消息体中携带的。
  8. URL:
    • RTSP使用URI请求时包含绝对URI。
    • 而由于历史原因造成的向后兼容性问题，HTTP/1.1只在请求中包含绝对路径，把主机名放入单独的标题域中；

总的来说，HTTP 主要用于静态内容的传输和网页浏览，而 RTSP 则专门用于控制实时流媒体的传输和播放。两者在设计和用途上有较大的差异，但都是在互联网中广泛使用的重要协议。

四、RTSP重要术语

  1. 集合控制（Aggregate Control）：
    集合控制是指在 RTSP 中，通过一个单独的 RTSP 控制连接来控制多个媒体流的行为。

    在一个会话中，可能会包含多个媒体流（如音频流、视频流等），通过集合控制，可以在一个控制请求中对多个媒体流进行操作，例如同时播放、暂停或停止。

  2. 实体（Entity）：
    实体通常指 RTSP 请求或响应中的消息主体，它包含了实际的数据或媒体内容。

    由以实体标题域（entity-header field）形式存在的元信息和以实体主体（entity body）形式存在的内容组成

    • 对于请求而言，实体可能是客户端发送的媒体数据；

    • 对于响应而言，实体可能是服务器返回的媒体数据或其他内容。

  3. 容器文件（Container File）：
    容器文件是指包含了一个或多个媒体流的文件，这些流可以是音频、视频或其他类型的数据。

    常见的容器文件格式包括 MP4、MKV、AVI 等，它们可以存储多种编码和格式的媒体数据，并提供了一种组织和管理这些数据的方式。

  4. RTSP 会话（RTSP Session）：
    RTSP 会话是RTSP交互的全过程。是指客户端与服务器之间建立的一种持久性的控制连接，用于控制实时流媒体的传输和播放。

    在 RTSP 会话中，客户端可以发送控制命令（如播放、暂停、停止等）给服务器，服务器则可以响应这些命令，并向客户端发送媒体数据或其他信息。会话通常由 RTSP 握手过程中的 SETUP、PLAY、PAUSE 和 TEARDOWN 等命令来管理。

    对一个电影的观看过程,会话(session)包括由客户端建立媒体流传输机制(SETUP)，使用播放(PLAY)或录制(RECORD)开始传送流，用停止(TEARDOWN)关闭流。

四、RTSP请求消息

  1. 消息格式：

方法 URI RTSP版本 CR LF

​ 消息头 CRLF CRLF

​ 消息体 CR LF

其中方法包括OPIONS、DESCRIBE、SETUP、PLAY、TEARDOWN等,URI是接受方的地址,例如:rtsp://192.168.0.1/video1.3gp。

RTSP版本一般都是 RTSP/1.0。每行后面的CR LF表示回车换行，需要接受端有相应的解析，最后一个消息头需要有两个CR LF

消息体是可选的，有的Request消息并不带消息体。

五、RTSP回应消息

  1. 消息格式：

    RTSP**版本状态码解释 CR LF

​ 消息头 CR LF CR LF

​ 消息体 CR LF

其中RTSP版本一般都是RTSP/1.0,状态码是一个数值，用于表示Request消息的执行结果,比如200表示成功,解释是与状态码对应的文本解释.

六、RTSP 重要方法

  1. OPTIONS:

用于得到服务器提供的可用方法；

如:

OPTIONS rtsp://192.168.20.136:5000/xxx666 RTSP/1.0

CSeq: 1

服务器的回应信息会在Public字段列出提供的方法。如:

RTSP/1.0 200 OK

CSeq: 1 //每个回应消息的cseq数值和请求消息的cseq相对应

Public: OPTIONS, DESCRIBE, SETUP, TEARDOWN, PLAY, PAUSE

  1. DESCRIBE****：

客户端向服务器端发送DESCRIBE，用于得到URI所指定的媒体描述信息，一般是SDP信息。客户端通过Accept头指定客户端可以接受的媒体述信息类型。

如：

C->S: DESCRIBE rtsp://server.example.com/fizzle/fooRTSP/1.0

CSeq: 312

Accept: application/sdp, application/rtsl,application/mheg)

服务器回应URI指定媒体的描述信息:

如:

S->C: RTSP/1.0 200 OK

CSeq: 312

Date: 23 Jan 1997 15:35:06 GMT

Content-Type: application/sdp //表示回应为SDP信息

Content-Length: 376

//这里为一个空行

//以下为具体的SDP信息

v=0

o=mhandley 2890844526 2890842807 IN IP4 126.16.64.4

s=SDP Seminar

i=A Seminar on the session description protocol

u=http://www.cs.ucl.ac.uk/staff/M.Handley/sdp.03.ps

e=mjh@isi.edu (Mark Handley)

c=IN IP4 224.2.17.12/127

t=2873397496 2873404696

a=recvonly

m=audio 3456 RTP/AVP 0

m=video 2232 RTP/AVP 31

m=whiteboard 32416 UDP WB

a=orient:portrait

媒体初始化是任何基于RTSP系统的必要条件，但RTSP规范并没有规定它必须通过DESCRIBE方法完成。RTSP客户端可以通过以下方法来接收媒体描述信息：

a) 通过DESCRIBE方法；

b) 其它一些协议（HTTP，email附件，等）；

c) 通过命令行或标准输入设备

  1. SETUP****：

用于确定转输机制，建立RTSP会话。客户端能够发出一个SETUP请求为正在播放的媒体流改变传输参数，服务器可能同意这些参数的改变。若是不同意，它必须响应错误”455 Method Not Valid In This State”。

Request中的Transport头字段指定了客户端可接受的数据传输参数；Response中的Transport 头字段包含了由服务器选出的传输参数。

如:

C->S: SETUP rtsp://example.com/foo/bar/baz.rm RTSP/1.0

CSeq: 302

Transport: RTP/AVP;unicast;client_port=4588-4589

服务器端对SETUPRequest产生一个Session Identifiers。

如:

S->C: RTSP/1.0 200 OK

CSeq: 302

Date: 23 Jan 1997 15:35:06 GMT

Session: 47112344 //产生一个SessionID

Transport: RTP/AVP;unicast;

client_port=4588-4589;server_port=6256-6257

  1. PLAY****：

PLAY方法告知服务器通过SETUP中指定的机制开始发送数据 。在尚未收到SETUP请求的成功应答之前，客户端不可以发出PLAY请求。

PLAY请求将正常播放时间（normal play time）定位到指定范围的起始处，并且传输数据流直到播放范围结束。PLAY请求可能被管道化（pipelined），即放入队列中（queued）；服务器必须将PLAY请求放到队列中有序执行。也就是说，后一个PLAY请求需要等待前一个PLAY请求完成才能得到执行。

比如，在下例中，不管到达的两个PLAY请求之间有多紧凑，服务器首先play第10到15秒，然后立即第20到25秒，最后是第30秒直到结束。

C->S: PLAY rtsp://audio.example.com/audio RTSP/1.0

CSeq: 835

Session: 12345678

Range: npt=10-15

C->S: PLAY rtsp://audio.example.com/audio RTSP/1.0

CSeq: 836

Session: 12345678

Range: npt=20-25

C->S: PLAY rtsp://audio.example.com/audio RTSP/1.0

CSeq: 837

Session: 12345678

Range: npt=30-

Range头可能包含一个时间参数。该参数以UTC格式指定了播放开始的时间。如果在这个指定时间后收到消息，那么播放立即开始。时间参数可能用来帮助同步从不同数据源获取的数据流。

不含Range头的PLAY请求也是合法的。它从媒体流开头开始播放，直到媒体流被暂停。如果媒体流通过PAUSE暂停，媒体流传输将在暂停点（the pause point）重新开始。

如果媒体流正在播放，那么这样一个PLAY请求将不起更多的作用，只是客户端可以用此来测试服务器是否存活。

  1. PAUSE****：

PAUSE请求引起媒体流传输的暂时中断。如果请求URL中指定了具体的媒体流，那么只有该媒体流的播放和记录被暂停（halt）。比如，指定暂停音频，播放将会无声。如果请求URL指定了一组流，那么在该组中的所有流的传输将被暂停。如：

C->S: PAUSE rtsp://example.com/fizzle/foo RTSP/1.0

CSeq: 834

Session: 12345678

S->C: RTSP/1.0 200 OK

CSeq: 834

Date: 23 Jan 1997 15:35:06 GMT

PAUSE请求中可能包含一个Range头用来指定何时媒体流暂停，我们称这个时刻为暂停点（pause point）。该头必须包含一个精确的值，而不是一个时间范围。媒体流的正常播放时间设置成暂停点。当服务器遇到在任何当前挂起（pending）的PLAY请求中指定的时间点后，暂停请求生效。如果Range头指定了一个时间超出了任何一个当前挂起的PLAY请求，将返回错误”457 Invalid Range” 。如果一个媒体单元（比如一个音频或视频禎）正好在一个暂停点开始，那么表示将不会被播放或记录。如果Range头缺失，那么在收到暂停消息后媒体流传输立即中断，并且暂停点设置成当前正常播放时间。

  1. TEARDOWN****：

TEARDOWN请求终止了给定URI的媒体流传输，并释放了与该媒体流相关的资源。如：

C->S: TEARDOWN rtsp://example.com/fizzle/foo RTSP/1.0

CSeq: 892

Session: 12345678

S->C: RTSP/1.0 200 OK

CSeq: 892

七、RTSP重要头字段参数

  1. Accept:

用于指定客户端可以接受的媒体描述信息类型。比如:

Accept: application/rtsl, application/sdp;level=2

  1. Bandwidth:

用于描述客户端可用的带宽值。

  1. CSeq****：

指定了RTSP请求回应对的序列号，在每个请求或回应中都必须包括这个头字段。对每个包含一个给定序列号的请求消息，都会有一个相同序列号的回应消息。

  1. Rang****：

用于指定一个时间范围，可以使用SMPTE、NTP或clock时间单元。

  1. Session:

Session头字段标识了一个RTSP会话。Session ID 是由服务器在SETUP的回应中选择的，客户端一当得到Session ID后，在以后的对Session 的操作请求消息中都要包含Session ID.

  1. Transport:

Transport头字段包含客户端可以接受的转输选项列表，包括传输协议，地址端口，TTL等。服务器端也通过这个头字段返回实际选择的具体选项。如:

Transport: RTP/AVP;multicast;ttl=127;mode=”PLAY”,

RTP/AVP;unicast;client_port=3456-3457;mode=”PLAY”

八、简单的RTSP消息交互过程

C表示RTSP客户端,S表示RTSP服务端

  1. 第一步：查询服务器端可用方法

1.C->S:OPTIONrequest //询问S有哪些方法可用

1.S->C:OPTIONresponse //S回应信息的public头字段中包括提供的所有可用方法

  1. 第二步：得到媒体描述信息

2.C->S:DESCRIBE request //要求得到S提供的媒体描述信息

2.S->C:DESCRIBE response //S回应媒体描述信息，一般是sdp信息

  1. 第三步：建立RTSP会话

3.C->S:SETUPrequest //通过Transport头字段列出可接受的传输选项，请求S建立会话

3.S->C:SETUPresponse //S建立会话，通过Transport头字段返回选择的具体转输选项，并返回建立的Session ID;

  1. 第四步：请求开始传送数据

4.C->S:PLAY request //C请求S开始发送数据

4.S->C:PLAYresponse //S回应该请求的信息

  1. 第五步： 数据传送播放中

S->C:发送流媒体数据 // 通过RTP协议传送数据

  1. 第六步：关闭会话，退出

6.C->S:TEARDOWN request //C请求关闭会话

6.S->C:TEARDOWN response //S回应该请求

上述的过程只是标准的、友好的rtsp流程，但实际的需求中并不一定按此过程。

其中第三和第四步是必需的！第一步，只要服务器客户端约定好，有哪些方法可用，则option请求可以不要。第二步，如果我们有其他途径得到媒体初始化描述信息（比如http请求等等），则我们也不需要通过rtsp中的describe请求来完成。

第二部分:SDP协议

一、SDP协议概述

SDP(SessionDescription Protocol )会话描述协议，用于描述多媒体会话，它为会话通知、会话初始和其它形式的多媒体会话初始等操作提供服务。

SDP的设计宗旨是通用性协议，所有它可以应用于很大范围的网络环境和应用程序，但 SDP 不支持会话内容或媒体编码的协商操作。

SDP信息包括：

  • 会话名称和目标；
  • 会话活动时间；
  • 构成会话的媒体；
  • 有关接收媒体的信息、地址等。

二、SDP格式

SDP 信息是文本信息，UTF-8 编码采用 ISO 10646 字符设置。SDP 会话描述如下（标注*符号的表示可选字段）：

  • v= （协议版本）
  • o= （所有者/创建者和会话标识符）
  • s= （会话名称）
  • i=* （会话信息）
  • u=* （URI 描述）
  • e=* （Email 地址）
  • p=* （电话号码）
  • c=* （连接信息 ― 如果包含在所有媒体中，则不需要该字段）
  • b=* （带宽信息）

　　一个或更多时间描述（如下所示）：

  • z=* （时间区域调整）
  • k=* （加密密钥）
  • a=* （0个或多个会话属性线路）
  • 0个或多个媒体描述（如下所示）

　　时间描述

  • t= （会话活动时间）
  • r=* （0或多次重复次数）

　　媒体描述

  • m= （媒体名称和传输地址）
  • i=* （媒体标题）
  • c=* （连接信息 — 如果包含在会话层则该字段可选）
  • b=* （带宽信息）
  • k=* （加密密钥）
  • a=* （0个或多个会话属性线路）

二、RTP

RTP头的结构体

CSRC

CSRCContributing Source标识符用于标识贡献源Contributing Sources

CSRC列表是一个可变长度的字段，用于指示参与当前RTP数据包的所有源。每个CSRC标识符都代表一个贡献源，通常是参与混合、合成或产生当前RTP数据包的源。这些贡献源可以是音频混合器中的不同参与者或者是参与多点会议的不同成员。

CSRC列表的存在使得接收者能够识别出哪些源对当前数据包做出了贡献，以便在接收端进行适当的处理，比如恢复丢失的数据、实现混音等操作。

每个CSRC标识符通常由32位组成，用于唯一地标识参与当前数据包的每个源。

在RTP头部中，CSRC列表的长度由RTP头部中的CSRC计数器字段指定，以便接收者知道CSRC列表的长度，并能够正确解析每个CSRC标识符。

Padding

  • Padding字段用于指示RTP数据包中是否存在填充字节。填充字节用于确保RTP数据包达到指定的长度，通常是为了满足某些传输协议对数据包长度的要求。
  • 如果Padding字段的值为1，则表明RTP数据包的末尾包含填充字节
  • 接收端在解析数据包时需要忽略这些填充字节。
  • 填充字节的内容可以是任意值，但接收端必须正确处理填充字节，以确保接收到的数据包内容是正确的。

Extension

  • Extension字段用于指示RTP数据包中是否包含扩展头部。扩展头部用于在RTP数据包中添加额外的信息，如时间戳、同步信号、参考信息等。
  • 如果Extension字段的值为1，则表明RTP数据包中包含扩展头部。
  • 扩展头部的具体格式由RTP头部中的Extension字段后面的一些字段定义，包括扩展头部的长度和扩展头部的内容。
  • 扩展头部的内容是可选的，可以根据需要进行定义和使用。接收端在解析数据包时需要根据扩展头部的格式来正确地解析和处理其中的信息。

Version

  • 该字段是一个2位的无符号整数，用于标识RTP协议的版本。当前版本的RTP协议是RTP版本2（RTPv2），其Version字段的值为2。
  • RTP的版本号可以用来识别和区分不同版本的协议，并确保接收端能够正确地解析和处理收到的RTP数据包。如果接收端检测到不支持的RTP版本号，通常会丢弃该数据包或者发送相应的错误消息。

Payload

  • Payload Type 字段用于指示 RTP 数据包中的有效载荷的类型，即数据的编码格式或者数据流的类型。
  • 例如，它可以指示音频数据的编码格式（如PCM、AAC、Opus等）或视频数据的编码格式（如H.264、VP9等）。
  • Payload Type 的范围是 7 位，可以表示 0 至 127 之间的整数。不同的 Payload Type 值对应着不同的数据类型或编码格式，这些值可以在 RFC 文档中找到相应的定义或约定。

Marker

  • Marker 字段是一个单独的比特位，用于向接收端传递额外的控制信息。
  • 它可以用来标记 RTP 数据流中的重要事件或特殊帧。
  • Marker 位的具体含义由应用程序或协议规范定义。
  • 例如，在视频传输中，Marker 位可能用于标记关键帧（I-frame）或其他重要的帧，以便接收端能够正确地处理数据流。

Seq

Sequence Number（序列号）

  1. 数据包排序：Sequence Number字段用于帮助接收端对RTP数据包进行排序。由于RTP数据包在网络传输过程中可能会出现乱序、丢失或延迟的情况，因此接收端可以通过检查数据包的序列号来重新排序数据包，以便按正确的顺序处理它们。
  2. 数据包丢失检测：接收端可以通过检查序列号来检测数据包是否丢失。如果接收端发现接收到的数据包的序列号与预期的序列号不匹配，就可以判断在传输过程中可能发生了数据包丢失。
  3. 同步：在多媒体流传输中，接收端可以利用序列号来保持与发送端的同步。通过跟踪序列号的变化，接收端可以准确地了解数据包的传输顺序和时序。

Sequence Number字段的初始值通常由发送端随机选择，以避免潜在的冲突。每发送一个RTP数据包，发送端会将序列号增加1。因此，接收端可以通过检测序列号的递增来确认数据包的顺序。

TimeStamp

Timestamp（时间戳）

  1. 时序同步：Timestamp字段允许接收端根据发送端的时钟信息来同步媒体流的播放。通过将采样时刻映射到时间戳，接收端可以根据时间戳来恢复发送端的采样时序，并确保媒体流的正确播放顺序。
  2. 时延计算：接收端可以利用时间戳来计算媒体流在传输过程中的时延。通过比较不同数据包的时间戳，接收端可以估算出数据包在网络中传输所经历的时间，并计算出媒体流的延迟情况。
  3. 时钟频率信息：Timestamp字段通常与RTP头部中的时钟频率信息一起使用，以确定时间戳的单位。时钟频率信息通常在RTP会话开始时通过SDP（Session Description Protocol）协商确定，用于将时间戳值映射到实际的时间单位（如毫秒或微秒）。

在实时音视频传输中，Timestamp字段对于确保媒体流的同步和时序非常重要。通过将采样时刻映射到时间戳，并利用时钟频率信息，接收端可以准确地恢复出发送端的采样时序，并实现同步的播放效果。

SSRC

SSRC（同步源标识符）是RTP协议（实时传输协议）头部的一个重要组成部分。它是一个32位的标识符，用于标识RTP数据包的源。在RTP通信过程中，SSRC的主要作用是识别各种同步源，以区分不同的媒体流。

SSRC的功能和重要性
  1. 流识别：SSRC确保在多流传输过程中，接收方可以区分来自不同源的数据包。例如，在一个视频会议中，可能有多个参与者同时发送音频和视频流，每个流都会有一个唯一的SSRC，接收方可以据此区分各个流。
  2. 冲突管理：在RTP会话中，如果发现SSRC冲突（即两个源偶然使用了相同的SSRC），参与者可以选择一个新的SSRC，以解决冲突。
  3. 会话管理和控制：在RTP控制协议（RTCP）中，SSRC用于关联媒体流与其控制信息，帮助执行如流量控制和监测等任务。
SSRC的生成

SSRC应当是随机生成的，以确保在广泛的网络环境中的唯一性和随机性。在RTP会话开始时，发送方生成一个随机的SSRC值。如果在会话中检测到SSRC的冲突，发送方需要生成一个新的SSRC值。

SSRC的设计确保了RTP能够支持复杂的多媒体会议、广播和其他实时通信应用，同时提供必要的流控制和同步功能。

四、其他协议

1. SDP

Session Description Protocol——会话描述协议

一种网络协议，用于描述多媒体通信会话的信息。

大白话——用文本描述的通信各端（PC 端、Mac 端、Android 端、iOS 端等）的能力。

这种协议不传输媒体本身，而是用于协商网络中两个或多个参与者之间的会话设置。SDP通常用于互联网电话（VoIP）、视频会议等应用中，它可以定义参与会话的媒体类型、数据传输协议、媒体流端口以及其他必要的配置信息。

SDP广泛应用在实时传输协议（RTP）中，特别是在实现语音和视频通话的应用中，比如Skype或者WebRTC技术。SDP通过在通信前提供必要的会话信息，使得参与者能够正确连接并交换媒体流。

关系

SDP和RTP的关系在于：

  • SDP通常用于描述会话中包含的媒体流的信息，包括媒体类型、编码格式、传输地址等。
  • RTP则负责实际传输媒体数据，但它需要依赖SDP提供的描述信息来确定媒体数据的类型、编码格式等参数，以便于正确解析和处理媒体数据。

SDP包括三个组件：

1、网关（Gateway）：负责保护业务系统，防护各类网络攻击，只允许来自合法客户端的流量通过；

2、客户端（Client）：负责验证用户身份，把访问请求转发给网关；

3、管控端（Controller）：负责身份认证、配置策略，管控整个过程。

SDP的5重防御机制：

（1）SPA（单包授权认证）：网关只有接收到了客户端发出的SPA包，验证合法之后，才对该客户端的IP开放指定端口。这个技术可以屏蔽绝大多数非法用户的网络攻击，让漏洞扫描、DDoS等攻击方式都失效。

（2）MTLS（双向认证）：网关跟客户端的通信是加密的，而且是双向认证，网关认证用户，用户也要认证网关。双向的认证保证了中间人攻击无法奏效。

（3）动态防火墙：经过SPA认证后，网关会放行指定端口。但端口放行是暂时的，几秒内没有操作，端口就会自动关闭。最大限度提高防护强度。

（4）设备验证：SDP不止验证用户身份，还要验证用户设备。设备的验证包括设备健康状态的验证，如是否装了杀毒软件等等。设备验证还包括设备证书的验证，只有合法设备才会安装证书，证书参与通信数据的加密过程。保证连接都是来自合法设备的。

（5）应用绑定：用户只能访问有授权的应用。这符合“最小化授权”原则，保证了威胁无法横向蔓延。

常见的DDoS攻击有三种：

（1）http攻击：与服务器建立大量http连接，让服务器耗尽精力，无力服务正常用户。

（2）Tcp syn攻击：通过向服务器发送大量syn包，让服务器始终处于等待状态。

（3）udp反射攻击：伪造udp请求发给第三方dns或ntp服务器，把udp包中的发起者改为攻击目标的IP，让第三方服务器反射并放大自己的攻击，对攻击目标造成大量冲击。

SDP针对三种攻击的防护机制分别是：

（1）http攻击：SDP根本不与非法用户建立http连接。

（2）Tcp syn攻击：SDP会把所有非法syn包直接丢掉，不会等待。

（3）udp反射攻击：SDP可以对一些非公开的dns或ntp进行保护，避免恶意udp包触达这些服务。

SDP适用场景总结

SDP架构需要用户安装客户端，所以SDP对终端的安全控制更强，客户端和网关联动能产生更严密的防护效果。但客户端也限制了SDP的应用场景。一些公开的网站没法用SDP保护。

SDP特别适用于合作伙伴、供应商、第三方人员、分支机构等等特定人群访问业务系统的场景。这些系统需要在互联网上开放，以便第三方人员访问。但是又不需要向全世界所有人开放，引来黑客攻击。这种场景下，SDP能保证合法用户正常连接，对未知用户“隐形”。SDP在这类场景下是相当能打的。
